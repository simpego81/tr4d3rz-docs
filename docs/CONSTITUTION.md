# Constitution for the tr4d3rz software company
Voglio completare il salto di paradigma: tr4d3rz deve diventare una software house gestita da agent Claude Code

## Ambiti di lavoro
### Repository
1. sono le directory tr4d3rz-* esistenti.
2. Ognuna è associata a uno o più technology device.
3. Gli agent lavorano qui per scrivere codice, debuggarlo, testarlo e attuare meccanismi di deployment

### Technology Device
1. Sono i dispositivi che esistono nell'ecosistema tr4d3rz e che fanno girare i software
2. Sono gestiti dagli agent e dalle pipeline per fare test e2e, demo, deployment, diagnostica, monitoraggio e statistiche.

### Roadmap
1. è il piano di sviluppo. 
2. Gli agent lavorano qui per definire un piano di sviluppo a seguito di feedback umani
3. deve essere incrementale e andare per complessità crescenti

### Documentation
1. La documentazione dell'ecosistema gira attorno al diagramma Archimate.
2. Ad esso sono correlati i diagrammi UML e la documentazione architetturale e di sviluppo.
3. Deve documentare l'ecosistema completo da specifica, annotando quali sono gli sviluppi fatti e quali ancora in piano.
4. Deve essere sempre aggiornata rispetto allo stato dell'arte.
5. deve essere navigabile mediante pagine github
6. le stakeholder view delle Github Pages devono attenersi al diagramma Archimate

## Agents
### team
1. Nel team degli agenti devono esserci almeno i seguenti ruoli:
- orchestratore
- planner
- architect
- reviewer
- developer
- debugger
- tester
- pipeline manager
- deployment manager
- process quality manager (che definisce e controlla che il processo, le regole e le modalità di lavoro siano conformi alla presente costituzione)

### Rules
1. Le regole per gli agent dovranno tener conto di questa costituzione.
2. il process quality manager si deve preoccupare che ci siano regole chiare per tutti gli agent e controllare che vengano rispettate
3. Le regole devono tener conto di mantenere lo stato coerente dei progetti, della pianificazione, delle specifiche e della documentazione
4. Il setup di collaborazione deve essere definito per risparmiare effort implementativo (e anche token), tenendo in considerazioni quali artefatti devono essere presenti per collaborare efficacemente. Può essere necessario prevedere diversi livelli di documentazione o artefatti per lavorare efficacemente nel contesto, evitando lavori ricorrenti.
5. ciascun ruolo o agent ha la sua lista di regole definita e il suo contesto chiaro sempre allineato
